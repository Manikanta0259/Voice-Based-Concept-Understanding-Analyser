# 5. Project Development Phase

## Purpose
Development notes, coding conventions, and implementation walkthrough referencing repository modules.

## Coding conventions & structure
- Keep single-responsibility per module: audio, ASR, NLP, scoring, reporting, DB.
- Use docstrings and type hints where helpful (existing modules have partial docstrings).
- Keep file I/O paths relative and configurable if moving DB out of source directory.

## Implementation walkthrough (key files)
- `source/app.py` — Streamlit app; main user flows:
  - authentication (database.register_user & authenticate_user)
  - concept CRUD (database.add_reference_concept)
  - upload, preview, analyze (audio_utils + speech_to_text + semantic_eval + scoring_engine)
  - persist & report (database.save_evaluation + report_generator.generate_pdf_report)

- `source/audio_utils.py` — audio loading, RMS/ZCR, silence detection and plotting
- `source/semantic_eval.py` — embeddings & keyword coverage logic
- `source/scoring_engine.py` — scoring composition and feedback messages
- `source/report_generator.py` — builds professional PDF reports

## Development notes & tips
- To iterate quickly on UI, run `streamlit run source/app.py` and use synthetic audio from `source/generate_samples.py`.
- Keep `uploads/` and `reports/` writable and add a cleanup task for CI or demo environments.
- When changing DB schema, update `init_db()` migration logic in `source/database.py`.

## Testing suggestions
- Add pytest tests under `tests/` for:
  - audio preprocessing (ensure consistent sr and normalization)
  - scoring_engine thresholds, filler detection
  - semantic_eval expected behaviour on contrived examples
  - database transaction correctness using a temporary DB path
